# Plan — Suggest Post Assignments (issue #197)

## Context

Today the planner assigns post members one at a time in PostsTab — opening each post, scanning matched members, picking one. With 18 posts and dozens of members, finding the globally optimal assignment (maximizing condition matches, avoiding duplicate-condition usage, respecting post priority) is tedious and error-prone.

This plan adds a **Suggest Post Assignments** feature: a button at the top of PostsTab opens a dialog showing an algorithm-generated proposal for every post, each with a checkbox. The planner can uncheck individual rows, then Apply commits the checked subset atomically.

Three architectural choices were locked in via clarifying questions:

1. **Hybrid apply path** — server stores the preview in `Siege` (mirrors autofill); apply delegates to existing `board_service.bulk_update_positions` for atomic commit. Avoids the partial-failure risk of the issue body's "N parallel updatePosition" approach while reusing tested infrastructure.
2. **First available position per post** — each post building has one group; the suggestion targets the first non-disabled, non-reserve position. If none available, the post is skipped.
3. **Top-of-tab toolbar + Dialog** — new toolbar above the post list with a "Suggest Assignments" button; opens a Dialog modeled after `DiscordSyncModal`.

GitHub Issue: [#197](https://github.com/glitchwerks/siege-web/issues/197) (existing, no new issue needed).

## Acknowledged limitations (per inquisitor pass 1)

The plan accepts these limitations explicitly so they don't surface as surprises during review:

- **Algorithm: priority-ordered greedy, not optimal.** The optimal solution to this problem is weighted bipartite matching (Hungarian algorithm). We deliberately use greedy because (a) `scipy` is not currently a backend dependency and adding it for one feature is over-budget, and (b) the planner reviews every suggestion and can uncheck rows, so suboptimality is recoverable. A documented known-suboptimal test case ships with the algorithm so the limitation is visible. Future work item if the suboptimality is observed in practice: switch to `scipy.optimize.linear_sum_assignment` (it's O(n³), trivially fast at 18 × ~30, and would give optimality guarantees).
- **Concurrency: single planner at a time.** Like the existing autofill, the preview lives in a column on the `siege` row and is therefore not keyed by user. If two planners preview the same siege within the TTL window, the second preview overwrites the first — both planners then see the second proposal on apply. The existing autofill has the same limitation; users have not reported pain. If multi-planner sessions become common, follow-up work would either (a) extract a `siege_preview` table keyed by `(siege_id, kind, user_id)` or (b) move ephemeral previews to Redis with the same key. Documented in the apply error message: "Preview was generated by another session — please regenerate."
- **Schema cost.** Two new JSON-and-datetime columns are added to `siege` mirroring autofill's pattern. We accept this minor table widening over the higher complexity of extracting a generic preview store. If a fourth preview pattern is ever proposed, that should be the trigger to refactor — not this feature.
- **No background TTL reaper.** Expired previews remain in the column until the next preview overwrites them or apply clears them. This matches autofill behavior. Net storage cost per siege is bounded (≤ a few KB JSON × 3 features); no cleanup job needed at current scale.

### 2. Service — algorithm

**New file:** `backend/app/services/post_suggestions.py`

Two public functions, both `async`, both follow the autofill `services/autofill.py:27-181` pattern:

```python
async def preview_post_suggestions(session: AsyncSession, siege_id: int) -> PostSuggestionPreviewResult
async def apply_post_suggestions(session: AsyncSession, siege_id: int, data: PostSuggestionApplyRequest) -> PostSuggestionApplyResult
```

**Algorithm (`preview_post_suggestions`):**

1. **Load eagerly** with `selectinload`: siege → posts → building → groups → positions; siege → siege_members → member → post_preferences; posts → active_conditions.
2. **Block if siege complete:** `if siege.status == SiegeStatus.complete: raise HTTPException(400, ...)` — matches `services/posts.py:61-62`.
3. **Collect candidate posts:** every Post on the siege whose building is not `is_broken`.
   - **Verified post structure:** posts in this codebase are 1-building × 1-group × 1-position (`backend/app/services/buildings.py:26`, `image_gen.py:100`). The plan does not generalize beyond this; if multi-position posts are ever introduced, this algorithm needs revisiting.
4. **For each post, identify the target position:** the single position belonging to the post.
   - **If `is_reserve=True` or `is_disabled=True` on that position:** emit a null-suggestion entry (`suggested_member_id=None`) with a `skip_reason` of `"reserve"` or `"disabled"`. Do **not** silently drop the post from the result. The UI shows the row with the reason so the planner can manually clear the flag and re-run if they want a suggestion. (Issue AC #6: "Posts with no matching member are left without a suggestion." We extend that to "no eligible target position." This is the explicit interpretation of charge #1.)
5. **Build candidate set per post:** active members of the siege whose `post_preferences` intersect the post's `active_conditions` (set intersection on `PostCondition.id`). If empty for a post, that post gets a null-suggestion row with `skip_reason="no_match"`.
6. **Sort posts by priority desc, then by `building.building_number` asc** (tiebreaker). Higher `Post.priority` (raw int) gets first pick — issue AC #3.
7. **Greedy assignment loop — explicit deterministic scoring contract:**
   - Track `used_member_conditions: dict[int, set[int]]` — member_id → set of post_condition_ids already suggested for that member elsewhere.
   - Track `member_assignment_counts: dict[int, int]` — total assignments per member, queried up front with this exact filter (charge #21):
     ```python
     SELECT Position.member_id, COUNT(*)
     FROM Position
     JOIN BuildingGroup ON Position.building_group_id = BuildingGroup.id
     JOIN Building ON BuildingGroup.building_id = Building.id
     WHERE Building.siege_id = :siege_id
       AND Position.member_id IS NOT NULL
       AND Position.is_disabled = False
       AND Position.is_reserve = False
       AND Building.is_broken = False
     GROUP BY Position.member_id
     ```
     Excludes disabled, reserve, and broken-building positions — only "live" assignments count toward load-balancing.
   - For each post in priority order, evaluate every candidate member with this **lexicographic** score tuple (lower is better):
     1. `duplicate_penalty` — `0` if the member has at least one matching condition not in `used_member_conditions[member_id]`, else `1`. Hard preference: a candidate whose match would not duplicate beats every candidate whose match would.
     2. `assignment_count` — `member_assignment_counts[member_id]`. Strict ordering by current assignment count, ascending. (Issue AC #5: soft signal — never a hard cap.)
     3. `member_name` — lexicographic ascending. Final tiebreak; guarantees same input → same output.
   - Pick the lowest-scoring candidate. For that member, pick the **matched condition** by:
     1. Prefer a condition in (member.preferences ∩ post.active_conditions) that is **not** already in `used_member_conditions[member_id]`; if multiple, pick the one with the **lowest `PostCondition.id`** (deterministic tiebreak).
     2. If all matching conditions are already used by this member, fall back to the lowest-id matching condition.
   - Record the suggested assignment; update both tracking dicts.
   - **Determinism contract:** Given the same `(posts, members, preferences, existing assignments)` snapshot, this algorithm produces byte-identical output across runs. Verified by determinism test in §5.
   - **Known suboptimality:** This is a greedy approximation, not optimal weighted bipartite matching. A documented suboptimal scenario ships in tests so the limitation is visible (see §5 test list). The planner reviews each suggestion and can uncheck — suboptimality is recoverable.
8. **Build result:** list of `PostSuggestionEntry` (one per post — including null-member entries with `skip_reason`). Include `position_id` (the target position picked in step 4) so the apply step has a direct write target. Include `current_member_id` and `current_member_name` from the position's existing assignment so the UI can show "current vs suggested."
9. **Persist preview:** `siege.post_suggest_preview = {"assignments": [...]}; siege.post_suggest_preview_expires_at = now + 30min; await session.commit()`. Identical TTL/shape to autofill. Note: a second preview within the TTL **overwrites** the first (single-planner-at-a-time limitation, accepted above).
10. **Return:** `PostSuggestionPreviewResult(assignments=..., expires_at=iso_string)`.

**`apply_post_suggestions` — with apply-time revalidation (race-fenced):**

This addresses inquisitor charges #4, #5, #13, #15, #17.

The whole sequence below runs in a **single SQLAlchemy transaction** (no intermediate `await session.commit()` between revalidation and write). The position SELECT in step 3 uses `select(...).with_for_update()` to acquire row-level locks on every target position, fencing the TOCTOU window between revalidation and `bulk_update_positions` (charge #13). Because `bulk_update_positions` itself does its own SELECT and commits, we **do not delegate** to it directly here — the apply path performs the writes inline using the same fields `bulk_update_positions` would have written, but inside the locked transaction. The reuse benefit is now smaller, but correctness wins.

1. Load siege + preview, validate TTL — raise 409 if expired/missing with detail `"No valid preview to apply, generate a new one"`.
2. Filter `raw_assignments = preview["assignments"]` to only entries whose `position_id` is in `data.apply_position_ids` AND has a non-null `suggested_member_id`. Entries with unknown `position_id` (not in the preview) are silently ignored.
3. **Re-fetch and lock target positions** with `select(Position).where(Position.id.in_(target_ids)).with_for_update()`. Joined to Building so we can also check `Building.is_broken`. For each entry, compare against the preview snapshot:
   - Position no longer exists → `stale_entries += {"position_id": id, "reason": "position_missing"}`.
   - `is_disabled` is now True → `"position_disabled"`.
   - `is_reserve` is now True → `"position_reserve"`.
   - **`Position.member_id != entry["current_member_id"]`** (charge #15): another planner wrote here since preview → `"member_changed"`. The frontend's "current vs suggested" comparison was based on stale data; the user must re-confirm against the new state.
   - `Building.is_broken` is now True → `"position_disabled"` (broken-building positions are functionally disabled).
4. **Re-validate suggested members** with `select(Member).where(Member.id.in_(member_ids))`. Inactive member → `"member_inactive"`.
5. **If `stale_entries` is non-empty:** raise 409 with detail `{"stale_entries": [{"position_id": int, "reason": str}, ...]}`. Transaction rolls back; zero positions written.
6. **Otherwise, write inline** (still inside the locked transaction). For each entry, set on the locked Position object (charge #17 — explicit field list):
   ```python
   position.member_id = entry["suggested_member_id"]
   position.is_reserve = False
   position.is_disabled = False
   position.matched_condition_id = entry["suggested_condition_id"]
   ```
   The matched-condition write is the load-bearing line — without it, AC #4 (avoid duplicate-condition usage) is invisible on the board.
7. Clear preview: `siege.post_suggest_preview = None; siege.post_suggest_preview_expires_at = None`.
8. **Single `await session.commit()`** — releases the FOR UPDATE locks atomically with the writes.
9. Return `PostSuggestionApplyResult(applied_count=len(written_entries))`.

**On `member_changed` resolution path:** the frontend's 409 banner already lists stale entries; the new reason gets a dedicated message ("Post 7 — another planner assigned a different member since you previewed"). "Regenerate preview" reloads — the user sees the up-to-date `current_member_id` next to the new suggestion and decides explicitly whether to overwrite.

**Divergences from `bulk_update_positions` (charge #24 — explicit inventory).** The original plan delegated apply to `bulk_update_positions`; the rewrite writes inline to fence the TOCTOU race (charge #13). Three behavioral differences are intentional and must be preserved:

1. **`_validate_position_state()` is not called.** `bulk_update_positions:227` validates the `(member_id, is_reserve, is_disabled)` triple. The inline path always writes `is_reserve=False, is_disabled=False, member_id=<int>` so the validator's invariants are vacuously satisfied. **Constraint:** any future widening of the inline writeset (e.g., a future "suggest reserve" feature) MUST re-introduce the call. This is enforced by a code comment on the inline write block referencing this section.
2. **`await session.refresh(pos)` is not called per position.** `bulk_update_positions:237-238` refreshes each updated position post-commit. The apply response only carries `applied_count` (no Position objects), so refresh staleness doesn't affect the response. **Constraint:** callers of `apply_post_suggestions` must re-query positions if they need fresh ORM state — they cannot trust session-resident objects from this code path. Tests that re-read state (e.g., the `matched_condition_id` propagation test in §5) already expire and re-query, which is the correct pattern.
3. **Inactive-member handling routes through `stale_entries` 409, not 400.** `bulk_update_positions:204-208` raises `HTTPException(400, "Member X is not active")`. The apply path classifies the same condition as `stale_entries[*].reason="member_inactive"` with status 409. This is intentional: the user's recovery path is "regenerate preview," not "fix unrelated input validation." **Constraint:** the 409 contract is part of the public API; tests assert on it.

The "reuses tested infrastructure" framing in pass-1 was traded for correctness in pass-2. This subsection makes the trade explicit so future maintainers don't silently re-introduce the delegation and lose the fence.

### 5. Backend tests

**New file:** `backend/tests/test_post_suggestions.py` — algorithm-level (mocked, fast).

Use the `SimpleNamespace` fixture style from `tests/test_autofill.py:28-89`.

**Issue AC coverage:**
- Single post, single member, member's preferences match post's conditions → suggestion matches that member.
- Two posts compete for the same member; higher-priority post wins (AC #3).
- One member overlaps multiple posts via the same condition → second post prefers a different matching condition for that member if available; if not, falls back to same condition (AC #4).
- Member already has 5 assignments; algorithm still suggests them but prefers less-loaded members first (AC #5 — soft, never blocking).
- Post with no matching member → entry has `suggested_member_id=None` and `skip_reason="no_match"` (AC #6).
- Post whose position is `is_reserve=True` → entry has `suggested_member_id=None` and `skip_reason="reserve"` (charge #1).
- Post whose position is `is_disabled=True` → `skip_reason="disabled"` (charge #1).

**Determinism + scoring contract (charges #2, #8):**
- Run preview twice on the same fixture; assert byte-identical output (after sorting any internal lists).
- Two members tie on duplicate-penalty + assignment-count; assert the lower-name member wins.
- Two conditions both match the post for the chosen member, neither used; assert the lower-id condition is picked.
- Member with 1 duplicate-condition assignment vs. member with 3 non-duplicate assignments → duplicate penalty wins (lexicographic tuple ordering).
- **Documented suboptimality — property-based, not exact-output (charge #14):** include a 2-post / 2-member fixture where greedy is known to under-fill. The test asserts **invariants** that any correct algorithm (greedy or improved) would satisfy, NOT the literal current output:
  - "no member is assigned the same condition to two posts simultaneously" (algorithm-agnostic correctness)
  - "every assigned post has a member whose preferences include the matched condition" (correctness)
  - "the number of assigned posts is ≥ 1" (greedy lower bound; an improved algorithm would push this to 2 and the test still passes)
  Pinning to the exact greedy output would turn the test into a tripwire that fails on the very algorithm improvement the plan endorses as future work.

**Apply behavior:**
- Apply with full `apply_position_ids` → writes all entries that have a non-null `suggested_member_id`; null-member entries are skipped (no error).
- Apply with subset of `apply_position_ids` → only those entries written.
- Apply with `apply_position_ids` containing IDs not in the preview → those IDs silently ignored (charge #9: contract test).
- Apply with `apply_position_ids=[]` → 0 writes, success (no-op).
- Apply on empty siege (no posts) → preview returns `assignments=[]`; apply with empty input → 0 writes (charge #9).
- Apply on siege with no members → preview entries all have `skip_reason="no_match"`; apply skips them (charge #9).
- Apply with expired TTL → 409 detail `"No valid preview to apply, generate a new one"`.
- Apply on completed siege → 400.
- Two consecutive previews within TTL → second overwrites first; apply uses second. **Lives in the integration suite, not the mocked suite (charge #16)** — must exercise real `await session.commit()` against the JSON column, not in-memory `SimpleNamespace` attribute reassignment. Assertions: after second preview, query the row and confirm `siege.post_suggest_preview` deserializes to the second result.

**Apply-time revalidation (charges #4, #5, #13, #15 — the load-bearing ones):**
- Preview generated → before apply, position flipped to `is_disabled=True` → apply 409 with `stale_entries=[{position_id, reason="position_disabled"}]`. Assert no positions written (atomic).
- Same with `is_reserve=True` → reason `"position_reserve"`.
- Building flipped to `is_broken=True` → reason `"position_disabled"` (broken building == functionally disabled).
- Member becomes inactive between preview and apply → reason `"member_inactive"`.
- **Position's `member_id` changed between preview and apply (charge #15)** → reason `"member_changed"`. Assert: no positions written; the user's apply does not silently overwrite the other planner's write.
- Multiple stale entries → all surfaced in single 409 response (one round-trip resolves all conflicts).
- **TOCTOU fence test (charge #13):** integration test that wraps revalidation + write in a transaction. Verify with a concurrent-update simulation: open two sessions, session A revalidates and is mid-transaction when session B tries to update the same position — session B blocks until session A commits or rolls back. (PostgreSQL `SELECT … FOR UPDATE` semantics; SQLite skipped.)
- **Concurrent apply on the same empty position (charge #22 — the central concurrency claim):** integration test. Two sessions both `previewPostSuggestions` against a siege with an empty post position; both attempt apply concurrently. Pin assertions:
  1. Exactly one apply returns 200 with `applied_count=1`.
  2. The other apply returns 409 with `stale_entries` containing exactly one entry, `reason="member_changed"`.
  3. Final `Position.member_id` equals the winning session's `suggested_member_id`.
  This is the load-bearing proof that the FOR UPDATE fence + `member_changed` revalidation actually compose correctly under contention. Without it, the apply rewrite is unverified for the most common write pattern (filling empty posts).
- **`matched_condition_id` propagation (charge #17):** apply commits → re-read the position from DB → assert `position.matched_condition_id == suggested_condition_id`. Without this assertion the duplicate-avoidance work the algorithm did is invisible in the field.

**New file:** `backend/tests/test_post_suggestions_integration.py` — DB-level (real test DB, slower).

Addresses inquisitor charge #12 — the algorithm depends on m2m eager-loading via `selectinload`. Mocked tests don't exercise that contract.

### 3. New component

**New file:** `frontend/src/components/PostSuggestionsModal.tsx`

Pattern: clone `frontend/src/components/DiscordSyncModal.tsx` shape (Dialog + DialogHeader + DialogContent + DialogFooter from `./ui/dialog`).

Internal state:
- `preview: PostSuggestionPreviewResult | null`
- `checked: Record<number, boolean>` — keyed by `position_id`
- `previewMutation` (TanStack `useMutation` calling `previewPostSuggestions`)
- `applyMutation` (calls `applyPostSuggestions` with the filtered `position_id` array)

Open behavior: on `open=true && !preview`, fire `previewMutation`; default checkbox state for each row is "checked" except where the suggestion has `suggested_member_id === null`.

Body: a table (use `./ui/table`) with columns:
| ☐ | Post # | Priority | Current | Suggested | |
|---|--------|----------|---------|-----------|---|
| Checkbox | building_number | priority badge | current_member_name + condition (or "—") | suggested_member_name + condition_description (or "no match found", muted) | "= current" tag if matches_current |

Sort rows server-side by priority desc + building_number asc (already done in algorithm). Disable checkbox + apply for rows with `suggested_member_id === null`.

Footer: `Apply Selected` (disabled if `applyMutation.isPending` or zero checked) + `Cancel`. On apply success, invalidate `["board", siegeId]` and `["posts", siegeId]`, close modal.

**409 stale-entries handling.** When the apply endpoint returns 409 with `detail.stale_entries`, the modal:
1. Stays open.
2. Renders an inline error banner above the table listing each stale entry by post number with its reason in plain English ("Post 7 — disabled since preview", "Post 12 — member became inactive").
